# TrendWatcher — SDG Lab MVP

Внутренний инструмент для анализа обсуждений на Reddit. Ежедневно выявляет новые темы, растущие тренды, усиливающиеся боли пользователей и формирует продуктовые гипотезы для основателей и маркетинга SDG Lab.

**Демо:** [sdg-lab-trendwatcher.netlify.app](https://sdg-lab-trendwatcher.netlify.app)

## Как это работает

Каждый день в 09:00 UTC Supabase pg_cron запускает Edge Function, которая:

1. **Собирает** горячие посты из настроенных сабреддитов (r/lonely, r/depression, r/socialskills) через 3-уровневую стратегию: Reddit OAuth → JSON API → RSS-фид (fallback)
2. **Анализирует** посты через OpenAI gpt-4o-mini со структурированным промптом, настроенным на извлечение actionable-сигналов
3. **Структурирует** результат в 4 категории сигналов: новые темы, растущие тренды, боли, продуктовые гипотезы
4. **Сохраняет** отчёт в Supabase PostgreSQL
5. **Отправляет** форматированный email-отчёт со ссылками на самые обсуждаемые посты через Brevo
6. **Отображает** отчёт на веб-дашборде со сравнением с предыдущим отчётом

Отчёты также можно генерировать вручную через кнопку «Generate Report» на дашборде.

```sql
create extension if not exists pg_cron schema pg_catalog;
create extension if not exists pg_net;

select cron.schedule(
  'daily-trendwatcher-report',
  '0 9 * * *',  -- каждый день в 09:00 UTC
  $$
  select net.http_post(
    url := 'https://<project-ref>.supabase.co/functions/v1/daily-report'::text,
    headers := '{"Content-Type": "application/json"}'::jsonb,
    body := '{}'::jsonb

  $$

```

Когда pg_cron запускается с пустым телом запроса, Edge Function читает сабреддиты и получателей email из таблицы `app_settings` — поэтому изменения, сделанные в UI, автоматически отражаются в ежедневных отчётах.

### Доступ к данным Reddit

Reddit блокирует API-запросы с IP-адресов облачных провайдеров. Edge Function использует 3-уровневый fallback:

1. **Reddit OAuth** (при наличии учётных данных) — официальный API с полным доступом к постам/комментариям
2. **JSON API напрямую** — работает локально, блокируется с большинства облачных IP
3. **RSS-фид** — `reddit.com/r/{sub}/hot.rss` — работает с любых IP, ограниченные метаданные, но достаточно для анализа на этапе MVP

Когда будут получены Reddit OAuth-данные (требуется одобренное Reddit-приложение), система автоматически переключится на полный API с более богатыми данными.

### AI-анализ: почему LLM, а не эмбеддинги

Ключевой вопрос: как извлечь actionable-сигналы из ~100 постов Reddit ежедневно?

**Вариант A: Эмбеддинги + кластеризация** — преобразовать каждый пост в вектор, кластеризовать похожие, измерять рост кластеров во времени. Даёт количественную точность («тема X выросла на 340% за 7 дней»), но требует: векторную БД (pgvector/Pinecone), пайплайн эмбеддингов, тюнинг кластеризации (min_cluster_size, epsilon), снижение размерности для визуализации и отдельный шаг для интерпретации кластеров в человекочитаемые сигналы.

**Вариант B: LLM-анализ (выбранный)** — передать посты в gpt-4o-mini со структурированным промптом. Модель читает, интерпретирует и выдаёт категоризированные сигналы за один проход.

Почему LLM выигрывает для MVP:

| Критерий | LLM | Эмбеддинги + кластеризация |
|---|---|---|
| Понимание сарказма, тона, подтекста | Да | Нет — «I love being alone /s» считается позитивным |
| Генерация продуктовых гипотез | Да — рассуждает о том, *что строить* | Нет — только группирует похожие посты |
| Время до запуска | Часы (промпт-инжиниринг) | Недели (инфра + тюнинг) |
| Инфраструктура | Один API-вызов | Векторная БД + пайплайн эмбеддингов + кластерные джобы |
| Стоимость за отчёт | ~$0.01 (gpt-4o-mini) | ~$0.005 эмбеддинги + хостинг БД |
| Сравнение между днями | Через Report Diff (UI) | Нативно (векторное сходство) |

**Правило 80/20:** LLM даёт 80% ценности (actionable-сигналы с контекстом) за 20% усилий. Эмбеддинги добавляют оставшиеся 20% ценности (количественная точность, детекция аномалий), но требуют 80% дополнительной инфраструктуры.

Промпт инструктирует модель:

- Обосновывать каждый сигнал конкретными паттернами постов (не расплывчатыми резюме)
- Связывать гипотезы с наблюдаемыми болями («Поскольку пользователи сообщают X, продукт, который Y, мог бы Z»)
- Калибровать силу по количеству постов: high (10+), medium (3–9), low (1–2)
- Оценивать процент роста на основе плотности постов и вовлечённости

### Сравнение отчётов

Когда существует 2+ отчёта, дашборд автоматически сравнивает выбранный отчёт с предыдущим и показывает:

- Новые сигналы, которых не было раньше
- Сигналы, которые усилились (сила выросла)
- Сигналы, которые ослабли
- Сигналы, которые больше не обнаруживаются
- Изменение объёма постов (абсолютное + процентное)

### Масштабирование за пределами MVP: эмбеддинги и кластеризация

Когда система накопит 30+ дней отчётов и команде потребуется количественная строгость, добавить:

**Эмбеддинги** (векторные представления смысла постов через `text-embedding-3-small`):

- **Кросс-дневное семантическое отслеживание** — «обсуждалась ли эта тема на прошлой неделе?» становится запросом на косинусное сходство, а не поиском по ключевым словам. Ловит перефразировки: «lonely in new city» / «moved, no friends» / «isolated after relocation» = один сигнал.
- **Детекция аномалий** — измерить, когда кластер темы вырос в 10 раз за 24 часа. LLM оценивает рост качественно; эмбеддинги измеряют математически.
- **Дедупликация** — посты из разных сабреддитов на одну тему группируются автоматически.
- **Инфраструктура:** pgvector в Supabase (уже доступен), ~$0.002/1K постов для генерации эмбеддингов.

**Кластеризация** (HDBSCAN / k-means по эмбеддингам):

- **Визуализация долгосрочных трендов** — графики эволюции тем за 30 дней. Видно, какие боли сохраняются, а какие — разовые всплески.
- **Обнаружение новых тем** — посты, не попадающие ни в один кластер = действительно новые сигналы. Надёжнее, чем суждение LLM «это действительно новое?»
- **Карты тематического ландшафта** — 2D-диаграммы рассеяния (через UMAP) показывающие, что обсуждают сообщества и как темы связаны друг с другом.
- **Инфраструктура:** Python-джоба (по расписанию или по запросу), HDBSCAN для кластеризации на основе плотности, UMAP для визуализации.

**Когда добавлять:** После накопления 30+ дневных отчётов в БД, когда команде нужно ответить на вопрос «как эта боль эволюционировала за последний месяц?» — вопрос, на который LLM-анализ сам по себе не может ответить надёжно.

## Формат отчёта

Каждый отчёт спроектирован как **короткий и actionable** — основатель должен прочитать его за 2 минуты и понять, что строить дальше.

### Структура

1. **Executive Summary** (2–3 предложения) — начинается с самого важного находки, не общих фраз
2. **Сигналы** по категориям (см. ниже) — каждый обоснован конкретными доказательствами из постов
3. **Сравнение отчётов** (только дашборд) — diff с предыдущим отчётом: новые/ушедшие/усилившиеся/ослабшие сигналы
4. **Самые обсуждаемые посты** (только email) — 15 постов с наибольшей вовлечённостью и прямыми ссылками на Reddit

### Категории сигналов

| Категория | Что отслеживает | Пример |
|---|---|---|
| 🆕 **Новые темы** | Темы, появляющиеся впервые или набирающие начальную тягу | «AI-компаньоны как эмоциональная поддержка в 3 ночи» |
| 📈 **Растущие тренды** | Темы, ускоряющиеся относительно базовой линии (с оценкой % роста) | «Discord-группы поддержки вместо терапевтических приложений (+45%)» |
| 😰 **Боли** | Конкретные фрустрации пользователей с неудовлетворёнными потребностями | «Пользователи жалуются, что терапевтические приложения "роботичные и шаблонные"» |
| 💡 **Продуктовые гипотезы** | Actionable-идеи, привязанные к болям выше | «Поскольку пользователи хотят человечной поддержки в нестандартное время, продукт, который матчит пиров по часовым поясам, мог бы снизить ночное одиночество» |

### Поля сигнала

Каждый сигнал включает:
- **Заголовок** — короткий (5–8 слов), описательный
- **Описание** — что происходит И почему это важно, со ссылками на наблюдаемые паттерны постов
- **Сила** — high (10+ постов с высокой вовлечённостью), medium (3–9), low (1–2, зарождающийся)
- **Сентимент** — positive / negative / mixed / neutral
- **Количество постов** — сколько постов подтверждают этот сигнал
- **Сабреддиты** — какие сообщества сгенерировали этот сигнал
- **Рост %** — оценка относительно обычного объёма (где применимо)

Email-отчёты дополнительно содержат ссылки на 15 самых обсуждаемых постов для быстрой верификации.

## Ключевые проектные решения

**Supabase как единый бэкенд.** Edge Functions, PostgreSQL, pg_cron — одна платформа вместо разброса по провайдерам. Netlify отдаёт только статику. Нет vendor lock-in на вычислительном уровне (Edge Functions — стандартный Deno, легко переносимый на Hono/Cloudflare).

**RSS-fallback для Reddit.** Reddit блокирует облачные IP при обращении к JSON API. Вместо блокировки используем RSS-фиды, которые работают с любых IP. Компромисс: меньше метаданных на пост, но достаточно для LLM-анализа. Путь обновления через OAuth готов в коде.

**LLM вместо эмбеддингов для MVP.** gpt-4o-mini по $0.15/1M входных токенов стоит ~$0.01 за отчёт. Даёт нюансированный анализ с actionable-гипотезами. Эмбеддинги + кластеризация отложены до момента, когда объём исторических данных оправдает инфраструктуру.

**Brevo для доставки email.** Выбран вместо Resend (ограничения на получателей в бесплатном плане) и Gmail SMTP (сложность авторизации в Deno). Бесплатный тариф Brevo — 300 писем/день на любой адрес без верификации домена — достаточно для внутреннего инструмента. REST API нативно работает в Deno Edge Functions. Получатели настраиваются из UI.

**UI, готовый для Storybook.** Все shared/ui-компоненты самодостаточны с типизированными пропсами — готовы к извлечению в пакет дизайн-системы.

**Настройки синхронизированы с БД.** Выбор сабреддитов и email-получателей сохраняется и в localStorage (мгновенный UI), и в таблице Supabase `app_settings`. Это гарантирует, что pg_cron использует те же настройки, что заданы в UI — без ручного изменения переменных окружения.

**Автоопределение dev/prod.** `VITE_FUNCTIONS_URL` в `.env.local` направляет запросы к локальным Edge Functions при разработке. Production-сборки игнорируют это автоматически (guard `import.meta.env.DEV`) — ручного переключения не требуется.

## Ответы на обязательные вопросы

### Какие сигналы наиболее ценны для SDG Lab?

1. **Неудовлетворённые потребности с эмоциональной срочностью** — когда пользователи выражают боль с сильными эмоциями вокруг общения, эмоциональной поддержки или коммуникационных барьеров. Это прямые сигналы продуктовых возможностей. Пример: «I wish there was an app where I could just talk to someone at 3am without judgment» — это спецификация, а не просто жалоба.

2. **Поведенческие сдвиги в механизмах совладания** — когда пользователи массово переходят на новые паттерны (AI-чатботы для одиночества, Discord-группы поддержки, анонимный пир-матчинг). Показывает органическое движение рынка, на котором SDG Lab может строить или улучшать.

3. **Негативный сентимент к существующим решениям** — фрустрация терапевтическими приложениями («слишком дорого», «feels robotic»), дейтинг-приложениями («усиливают одиночество»), соцсетями («ловушка сравнения»). Раскрывает пробелы для дифференцированного позиционирования и конкретные требования к фичам.

### Как сделать это конкурентным преимуществом?

- **Скорость:** ежедневные сигналы vs квартальные исследования. Обнаружить боль в понедельник, описать MVP во вторник, зарелизить к пятнице. Конкуренты проводят ежегодные UX-исследования.
- **Доменная настройка:** промпты, выбор сабреддитов и таксономия сигналов настроены именно под вертикали SDG Lab (одиночество, эмоциональная поддержка, p2p-коммуникация). Универсальные тренд-инструменты упускают нюансы.
- **Историческая память:** лонгитюдный датасет ежедневной эволюции болей становится конкурентным рвом со временем — видно не только сегодняшние проблемы, но и как они меняются помесячно.
- **Пайплайн «гипотеза → бэклог»:** каждый вывод actionable («Поскольку пользователи сообщают X, постройте Y»), а не информационный. Продуктовые решения напрямую следуют из данных.

### Как автоматизировать пайплайн «тренд → MVP»?

1. **Фаза 1 (текущая):** Структурированные ежедневные отчёты → человек читает, приоритизирует, создаёт тикеты
2. **Фаза 2:** Система скоринга `opportunity = strength × domain_fit × competition_gap × urgency` → автоматически ранжированные гипотезы в Slack с приоритетными баллами
3. **Фаза 3:** AI-генерация продуктовых спецификаций из одобренных гипотез — персоны, user stories, скоуп MVP, эскиз технической архитектуры
4. **Фаза 4:** Полный цикл: обнаружить сигнал → оценить → сгенерировать спеку → создать репо из шаблона → назначить команду. Человеческое одобрение только на контрольных точках.

## Планы на будущее

- **Аутентификация и роли** — сейчас дашборд публичный. Добавить Supabase Auth (email/magic link) с ролями: `admin` (полный доступ, настройки, генерация отчётов), `viewer` (только просмотр дашборда). RLS-политики на таблицах `reports` и `app_settings` должны ограничивать доступ только для аутентифицированных пользователей. Edge Function должна валидировать JWT при ручных триггерах.
- **Страница аналитики** — графики на Recharts прямо в дашборде: объём постов за 30 дней, распределение категорий сигналов, тепловая карта повторяющихся тем. Все данные уже есть в таблице `reports` Supabase.
- **Эмбеддинги + кластеризация** — pgvector для кросс-дневного семантического сходства, HDBSCAN для долгосрочного обнаружения тем (см. [Масштабирование за пределами MVP](#масштабирование-за-пределами-mvp-эмбеддинги-и-кластеризация))
- **Slack-интеграция** — доставка ежедневных отчётов в Slack-канал параллельно с email
- **Reddit OAuth + анализ комментариев** — полный доступ к API при получении одобрения; позволит загружать топ-комментарии к постам для более глубокого анализа сентимента (сейчас анализируются только заголовки и текст постов, комментарии не загружаются из-за ограничений RSS)
- **Скоринг возможностей** — автоматическое ранжирование гипотез по `strength × domain_fit × competition_gap × urgency`
- **Мониторинг ошибок** — Sentry для фронтенда + алертинг Edge Function
